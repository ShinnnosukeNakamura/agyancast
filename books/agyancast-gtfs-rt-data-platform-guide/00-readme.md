この本のテーマは「**地域の混雑（= 周辺道路の詰まり）を、天気予報みたいに見たい**」です。

週末のショッピングモール周辺、通勤ピークの幹線道路、イベント開催日の市街地。
「今日はどこが詰まりそうか」を知れれば、出発時間や行き先をずらしてストレスを減らせます。

でも現実には、**渋滞の情報を“細かく・広い範囲で・高頻度に”集め続ける**のは難しいです。
Googleマップを見れば“いま”は分かりますが、次のような用途には向きません。

- 地域全体を面で監視して、毎日・毎週の傾向をためる
- 何ヶ月分も履歴を持って「比較」や「予報」の材料にする
- 個人開発・小規模運用で、コストを読みながら継続する

そこで本書では、渋滞そのものを直接集めるのではなく、**継続取得しやすいオープンデータを使って“混雑の気配”を観測する**アプローチを取ります。

題材にするのは、`agyancast` という実プロジェクトです。
GTFS-RTを「システムとして動かす」ために必要な流れ（収集→保存→変換→配信）を、実装と一緒に順番に追います。

## 使うデータ：GTFS / GTFS-RT とは

- **GTFS（Static）**：停留所・路線・時刻表など、交通の「地図と基準」
- **GTFS-RT（Realtime）**：遅延・運行状態など、交通の「いま起きている変化」

本書の中心は **GTFS-RT** です。
バスの遅延は、地域の混雑（道路の詰まり）と関係があることが多く、しかも **継続的に取得しやすい**。
この「扱いやすい観測データ」を積み上げて、地域の混雑を“見える化”する土台を作ります。

## この本がゴールとするもの

この本は「高精度な渋滞推定モデル」を作る本ではありません。
まずは、次の状態まで到達するのがゴールです。

- GTFS-RTを定期的に取得して、履歴として蓄積できる
- ためたデータを後から検索・集計できる（= 比較できる）
- “混雑っぽさ”を見える形にして、体験として成立させる

予報（未来の推定）は、その上に乗る「次の一手」として扱います。

## 先に結論

このプロジェクトで一番重要だった判断は、次の3つです。

1. **変換をやり直せるように、取得したGTFS-RT（BIN）をそのまま失われない形で保管する**（変換前の元データ）
2. **集計とデバッグの土台として、必要項目を抜き出してイベントログとして蓄積する**（1観測=1レコード）
3. **使い方が違うので、画面向けのスナップショットと分析向けの履歴・集計を分けて出力する**（最新 vs 時系列）

つまり、「いきなり最適解を目指す」のではなく「壊れにくい土台から積む」方針です。

ここで出てくる用語は、後の章でちゃんと説明します。

- **BIN（protobuf）**：GTFS-RTが配られてくるバイナリ形式（中身は後でJSONに展開します）
- **JSONL**：1行が1つのJSONになっているログ形式（イベントログに向いています）
- **スナップショット**：いまの状態を1つにまとめたファイル（例: `latest.json`）

## 読み終えた時のゴール

読み終えた時に、次を自分の言葉で説明できれば成功です。

- 「渋滞データを直接集める」ことが、なぜ運用として重いのか
- GTFS / GTFS-RTで何が取れて、どこに限界があるのか
- GTFS-RTをAWS上で蓄積・検索できる形にする基本方針
- GTFS static と GTFS-RT の違い
- 今回の仕様で「どのファイルのどの列」を使っているか
- BINデータをどう取り込み、どうJSONL/Parquetへ変換しているか
- 配信（最新）と分析（履歴・集計）を分けておくと、なぜ運用が楽になるのか
